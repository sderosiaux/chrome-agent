# dev-browser v2 — Design Document

## Problem Statement

dev-browser v1 automates browsers for AI agents via sandboxed JavaScript scripts. It works — benchmarks show it beats Playwright MCP (29 turns/$0.88 vs 51 turns/$1.45). But the architecture carries enormous weight:

- **76K lines of code**, of which 69K are a forked Playwright client that must be manually rebased on every upstream release
- **Node.js daemon** running permanently in the background
- **Runtime dependencies**: Node.js + npm + Playwright + quickjs-emscripten
- **~200 MB installed footprint** (Node modules + Chromium tooling)
- **~500ms startup** (daemon health check + IPC)
- **QuickJS WASM sandbox** adding indirection: script → QuickJS → forked Playwright client → JSON bridge → host Playwright → CDP → Chrome

The fork is 90% of the codebase and exists solely to make Playwright's client run inside QuickJS WASM. Agents use ~17 methods of the Playwright API. The rest is dead weight.

## Vision

Single Rust binary. Zero runtime dependencies. The agent sends JS that runs directly in Chrome via CDP. No Node.js, no Playwright, no QuickJS, no WASM, no daemon, no fork.

```
v1: Script → QuickJS WASM → Forked Playwright → Host Playwright → CDP → Chrome
v2: Script → Rust CLI → CDP → Chrome
```

## Design Decisions

### D1: Eliminate the Playwright fork by talking CDP directly

Chrome DevTools Protocol provides everything agents need: navigation, DOM interaction, screenshots, accessibility tree, JS evaluation. Playwright and Puppeteer are convenience layers over CDP. For the ~17 methods agents use, a thin CDP client in Rust replaces 69K lines of forked code.

Validated by: chrome-devtools-mcp (Google, 32K stars) uses the same approach — Puppeteer over CDP with a11y tree snapshots and uid-based element targeting.

### D2: Eliminate QuickJS WASM — Chrome IS the sandbox

v1 runs scripts in QuickJS WASM to prevent host access. But when executing browser automation scripts, the natural execution environment is Chrome itself. Chrome's sandbox already prevents filesystem/process access. `Runtime.evaluate` via CDP executes JS in the page context with a configurable timeout.

Trade-off: scripts can `fetch()` from the page context. This is intentional — agents need web access. For untrusted third-party code, an optional isolated world via `Runtime.evaluate({contextId})` can be added later.

### D3: Replace the Node.js daemon with an optional Rust micro-daemon

The v1 daemon exists to keep browsers and named pages alive between invocations. The problem was never the daemon pattern — it was the Node.js + Playwright weight behind it. v2 offers two modes:

**Stateless mode (default for one-shot commands)**:
1. Chrome launched with `--remote-debugging-port` stays alive independently of the CLI
2. A JSON session file (`~/.dev-browser/sessions.json`) tracks browser WebSocket endpoints, PIDs, and named page targetIds
3. Each CLI invocation connects via WebSocket, acts, disconnects

**Daemon mode (auto-started for sessions, `dev-browser daemon start`)**:
1. A micro-daemon in Rust (~500 lines) runs as a background process
2. Holds a persistent WebSocket connection to Chrome — eliminates reconnection overhead per invocation
3. Monitors Chrome health via heartbeat — detects crashes, cleans up orphaned sessions
4. Manages state in-memory — no file-locking race conditions on concurrent CLI invocations
5. CLI communicates with daemon via Unix socket (macOS/Linux) or named pipe (Windows)
6. Auto-started on first `goto`/`snap`/`click` if not running. Auto-exits after idle timeout (5 min).

The daemon is a single Rust binary, not a separate executable — `dev-browser` itself is the daemon when invoked as `dev-browser daemon start`. ~2 MB memory footprint.

**Why both modes**: One-shot `eval` or `screenshot` commands don't need a daemon. Multi-step agent sessions (snap → click → snap → fill → snap) benefit from persistent connection, heartbeat, and in-memory state.

### D4: uid-based element targeting instead of CSS selectors

Inspired by chrome-devtools-mcp. The a11y tree snapshot assigns a unique `uid` to each element, mapped to a CDP `backendNodeId`. The agent reads the snapshot, picks a uid, sends the action. No CSS selector guessing, no fragile XPath.

```
Snapshot output:
  uid=e12 button "Sign In" focusable
  uid=e13 textbox "Email" value=""
  uid=e14 textbox "Password" value=""

Agent action:
  dev-browser click main e12
```

Resolution path: uid → `ElementRef` (abstract) → resolved via CDP at action time.

### D4a: ElementRef abstraction — no direct CDP lock-in

uids do not map directly to `backendNodeId` in the session file. Instead, they map to an `ElementRef` that encapsulates the resolution strategy:

```rust
enum ElementRef {
    BackendNode { backend_node_id: u64 },
    // Future: BiDi, objectId, selector fallback, etc.
}
```

The session file stores:
```json
"uidMap": {
  "e1": { "type": "backendNode", "id": 47 },
  "e2": { "type": "backendNode", "id": 52 }
}
```

This indirection prevents lock-in to CDP internals. If Chrome deprecates `backendNodeId`, changes the Accessibility API, or the ecosystem moves to WebDriver BiDi, only the `ElementRef` resolver changes — not the session format, not the CLI surface, not the agent-facing uids. Playwright served as this buffer in v1; `ElementRef` is the v2 equivalent, but at ~20 lines of Rust instead of 69K lines of TypeScript.

### D5: Three-tier interaction model (phased delivery)

Agents need both speed (single actions) and power (multi-step scripts). Three tiers cover the full range:

**Tier 1 — Atomic commands** (90% of interactions) — **v2.0**
One CLI call, one action, one result. The agent generates one line. Minimum tokens. With `--snap`, the agent gets the updated page state in the same call — effectively batching action + observation.

**Tier 2 — Eval** (DOM extraction, complex queries) — **v2.0**
JS executed in Chrome via `Runtime.evaluate`. The browser runs it. No external sandbox.

**Tier 3 — Batched scripts** (multi-step orchestration) — **v2.1**
Multiple actions in one invocation via `Runtime.addBinding` bridge. Deferred because:
1. Tier 1 + `--snap` already covers 90%+ of agent interactions with minimal tokens
2. The `Runtime.addBinding` bridge (`goto` destroys page context) is architecturally complex — comparable to the QuickJS bridge it replaces
3. Shipping Tier 1+2 first validates the CDP-direct approach with real agents before committing to Tier 3 complexity

Batching is dev-browser's competitive advantage (29 turns vs 51 for Playwright MCP). v2.0 achieves most of this via `--snap` on atomic commands. v2.1 adds full script mode for the remaining edge cases.

### D6: Helper runtime injected into Chrome (v2.1)

Deferred to v2.1 alongside Tier 3 script mode.

~200 lines of JS, embedded in the Rust binary via `include_str!`, injected into the page context before Tier 3 scripts. Provides Playwright-like DX without Playwright:

- `waitForSelector(selector, {timeout})` — poll-based retry
- `click(selectorOrUid)` — wait + click with retry
- `fill(selectorOrUid, value)` — focus + set value + dispatch events
- `snap()` — request a11y tree snapshot (delegates to Rust via CDP binding)
- `waitForURL(pattern, {timeout})` — poll location.href
- `goto(url)` — delegates to Rust via `Runtime.addBinding` (avoids page context destruction)

The `Runtime.addBinding` bridge pattern enables CDP-level operations from within scripts. See implementation plan M7 (v2.1) for the full bridge design.

### D7: snapshotForAI via CDP Accessibility.getFullAXTree

The feature that saves the most tokens. Implementation:

1. CDP `Accessibility.getFullAXTree` returns the full accessibility tree with roles, names, values, states, and `backendNodeId` per node
2. Rust parses and formats it into a compact text representation with uid identifiers
3. A `Map<uid, backendNodeId>` is stored in the session for subsequent actions

Output format (matches chrome-devtools-mcp convention):
```
uid=e1 heading "Welcome back" level=1
uid=e2 textbox "Email" value="" focusable
uid=e3 textbox "Password" value="" focusable
uid=e4 button "Sign In" focusable
uid=e5 link "Forgot password?"
```

~50 tokens vs ~2000 tokens for raw HTML. This is the core token optimization.

## CLI Surface

### Atomic commands (Tier 1)

```bash
dev-browser goto <page> <url>
dev-browser click <page> <uid>
dev-browser fill <page> <uid> <value>
dev-browser snap <page> [--verbose]
dev-browser screenshot <page> [filename]
dev-browser eval <page> <expression>
dev-browser tabs
dev-browser close <page>
dev-browser status
dev-browser stop
dev-browser install                    # download Chromium
```

### Eval (Tier 2 — v2.0)

```bash
dev-browser eval main "document.title"
dev-browser eval main <<'EOF'
  [...document.querySelectorAll('.price')].map(e => e.textContent)
EOF
```

### Script mode (Tier 3 — v2.1)

```bash
dev-browser run main <<'EOF'
  await goto("https://shop.com");
  const s = await snap();
  await click("e12");
  await fill("e13", "user@test.com");
  await click("e14");
  return await snap();
EOF
```

Deferred to v2.1. In v2.0, agents achieve the same by chaining atomic commands with `--snap`.

### Batch fill

```bash
dev-browser fill-form main e2="user@test.com" e3="password123" e4="John"
```

One command, multiple fields. Reduces round-trips for form-heavy flows. Inspired by chrome-devtools-mcp `fill_form`.

### `--snap` flag on atomic commands

Every action command accepts `--snap` to return a fresh snapshot after the action completes. Saves one round-trip per interaction:

```bash
# Without --snap: 2 calls
dev-browser click main e12
dev-browser snap main

# With --snap: 1 call
dev-browser click main e12 --snap
→ Successfully clicked
→ uid=e1 heading "Dashboard" ...
```

### Global flags

```bash
--browser <name>         # named browser instance (default: "default")
--connect [url]          # attach to running Chrome (auto-discover or explicit)
--headless               # launch Chromium headless
--timeout <seconds>      # script/command timeout (default: 30)
--ignore-https-errors    # accept self-signed certs (v1 parity)
```

## Session File

```json
{
  "default": {
    "wsEndpoint": "ws://127.0.0.1:9222/devtools/browser/abc-123",
    "pid": 12345,
    "headless": false,
    "daemonPid": null,
    "pages": {
      "main": {
        "targetId": "ABCDEF1234567890",
        "uidMap": {
          "e1": { "type": "backendNode", "id": 47 },
          "e2": { "type": "backendNode", "id": 52 },
          "e3": { "type": "backendNode", "id": 58 }
        }
      }
    }
  }
}
```

The `uidMap` uses `ElementRef` objects (not raw integers) so the resolution strategy can evolve without breaking the session format. `daemonPid` tracks the optional micro-daemon process; `null` means stateless mode.

The uidMap is refreshed on every `snap` command. Between snaps, cached uids are used for actions. If a uid resolves to a detached node, the CLI returns an actionable error suggesting a fresh snapshot.

## Security Model

Different threat model from v1. v1 protects against malicious scripts accessing the host. v2 scripts run in Chrome — Chrome's sandbox is the boundary.

| Vector | v1 (QuickJS WASM) | v2 (Chrome context) |
|---|---|---|
| Host filesystem | Blocked by WASM | Blocked by Chrome sandbox |
| Arbitrary network | Blocked by WASM | Page can `fetch()` — intentional |
| Launch processes | Impossible | Impossible (Chrome sandbox) |
| Cross-tab access | Host bridge isolation | Same-origin + CDP target isolation |
| Infinite loop | QuickJS CPU interrupt | `Runtime.evaluate` timeout parameter |
| Memory | 512MB QuickJS limit | Chrome manages |

For adversarial scenarios (untrusted third-party scripts), a future option could use `Page.createIsolatedWorld` via CDP to run scripts in an isolated JS context that cannot access the page's JS objects.

Reduced attack surface: no fork to maintain, no daemon with Unix socket, no Node.js process.

## Network Model

v1 has three network layers: CLI ↔ daemon (Unix socket IPC), daemon ↔ Playwright (in-process), Playwright ↔ Chrome (CDP over pipe or WebSocket). v2 collapses this to one: CLI ↔ Chrome (CDP over WebSocket).

```
v1: CLI ──Unix socket──► Node daemon ──in-process──► Playwright ──CDP──► Chrome
v2: CLI ──WebSocket CDP──► Chrome
```

WebSocket works identically on macOS, Linux, and Windows. No platform-specific IPC (Unix sockets vs Windows named pipes). The only platform-specific code is Chromium path discovery for auto-connect.

## Error Handling — Self-Healing Errors

Inspired by chrome-devtools-mcp design principle: errors include context and a suggested fix.

| Error | Message |
|---|---|
| Stale uid | `Element uid=e12 no longer exists. The page may have changed. Run 'dev-browser snap main' to get fresh uids.` |
| No snapshot | `No snapshot for page "main". Run 'dev-browser snap main' first to discover elements.` |
| Browser dead | `Browser "default" (pid 12345) is no longer running. Run 'dev-browser goto main <url>' to start a new session.` |
| Timeout | `Script timed out after 30s. Use --timeout 60 for longer operations.` |
| Element not interactable | `Element uid=e5 is not visible or is covered by another element. Try scrolling or waiting.` |
| No Chrome found | `Could not find Chrome with remote debugging. Launch Chrome with --remote-debugging-port=9222 or run 'dev-browser install' for a managed Chromium.` |

## Action Stabilization — waitForEventsAfterAction

After click/fill actions, the page may trigger navigations, XHR requests, or DOM mutations. The CLI waits for the page to stabilize before returning:

1. Dispatch the action (click, fill, etc.)
2. Wait up to 500ms for `Page.frameNavigated` or `Page.loadEventFired` events
3. If a navigation started, wait for it to complete (up to `--timeout`)
4. If no navigation, return immediately after action dispatch

This prevents the agent from snapping a mid-transition page. chrome-devtools-mcp uses the same pattern via `waitForEventsAfterAction`.

## Screenshots — Reference over Value

Screenshots return a file path, not base64 on stdout. This saves tokens (a base64 screenshot is ~100K tokens) and lets multimodal agents read the file directly.

```bash
dev-browser screenshot main
→ /Users/me/.dev-browser/tmp/screenshot-main-1711540200.png

dev-browser screenshot main login-page.png
→ /Users/me/.dev-browser/tmp/login-page.png
```

File I/O restricted to `~/.dev-browser/tmp/` (same as v1).

## Browser Profile Persistence

Chromium launched by the CLI uses a persistent user data directory at `~/.dev-browser/browsers/<name>/chromium-profile`. This preserves cookies, localStorage, and session data across invocations — the agent can log in once and stay logged in.

```bash
dev-browser goto main https://app.example.com    # logs in
# ... later invocation ...
dev-browser eval main "document.cookie"           # still logged in
```

Same behavior as v1's `launchPersistentContext`.

## Cross-Platform

| Concern | macOS | Linux | Windows |
|---|---|---|---|
| CDP transport | WebSocket | WebSocket | WebSocket |
| DevToolsActivePort paths | `~/Library/Application Support/Google/Chrome/` | `~/.config/google-chrome/` | `%LOCALAPPDATA%\Google\Chrome\User Data\` |
| Chromium launch | Direct exec | Direct exec | `chrome.exe` |
| Session file | `~/.dev-browser/sessions.json` | same | `%USERPROFILE%\.dev-browser\sessions.json` |
| Named pipes / Unix sockets | Not needed (WebSocket) | Not needed | Not needed |

v1 needed platform-specific IPC (Unix sockets vs Windows named pipes). v2 uses WebSocket everywhere — the only platform code is path resolution.

## What We Lose

1. **Full Playwright API** — no `locator().filter().nth()`, no multi-frame auto-detection, no sophisticated auto-waiting. The helpers cover 95%. The remaining 5% uses raw `eval`.

2. **Strict sandbox** — scripts can `fetch()`, read page cookies, manipulate page DOM. For agent-controlled use, this is fine. For untrusted code, needs isolated world.

3. **Script/page isolation** — v1 scripts run in QuickJS, separate from the page. v2 scripts run in the page context. If the page redefines `document.querySelector`, the script could be affected. Mitigated by using `Runtime.evaluate({contextId})` with an isolated world.

## Prior Art

- **chrome-devtools-mcp** (Google, 32K stars): MCP server with Puppeteer, a11y snapshot, uid-based targeting. Validates the uid model and CDP approach. Does not batch actions — each MCP call is one action.
- **Puppeteer**: Google's CDP automation library. Our CDP client replaces the subset we need (~20 CDP methods).
- **dev-browser v1**: Proves that batching and persistent pages reduce agent cost by 40%+ vs one-action-per-call approaches.

## Open Questions

1. **Chromium download**: bundle a download command (`dev-browser install`) or rely on system Chrome? Recommend: both. Auto-discover system Chrome, offer `install` for headless CI.

2. **MCP mode**: should v2 also expose an MCP server interface for agents that prefer MCP over CLI? Recommend: add later as a thin wrapper over the same core. CLI-first.

3. **Incremental snapshots**: chrome-devtools-mcp sends full snapshots each time. A diff-based approach could further reduce tokens for repeated snaps. Recommend: defer to v2.1.
